# 1. Requirements Definition (要件定義書)

## 1.1. 概要 (Overview)
本ドキュメントは、Minecraft Mod "Hyper-Visor Storage" の機能要件および非機能要件を定義する。
本システムは、Rust (Wasm) と Java のハイブリッドアーキテクチャを用い、従来のModでは達成不可能な超高性能物流・ストレージシステムを提供することを目的とする。

---

## 1.2. 機能要件 (Functional Requirements)

### F-01: コアストレージ機能 (Core Storage)
*   **F-01-01 大容量管理:** 1アイテム種につき最大 `u64::MAX` (約1844京) 個まで格納可能であること。
*   **F-01-02 種類数:** システム全体で `u32::MAX` (約42億) 種類のアイテムIDを識別可能であること。
*   **F-01-03 NBTサポート:** エンチャントや耐久値を含むアイテムをハッシュ化して区別し、個別に格納可能であること。
*   **F-01-04 永続化:** サーバー停止時および定期バックアップ時に、Wasmメモリ状態をバイナリ形式でディスクに保存し、再起動時に復元できること。

### F-02: 輸送・物流機能 (Transport Logic)
*   **F-02-01 チャンネル制:** 物理的なパイプ接続に依存せず、「チャンネルID」を用いた論理接続でアイテムを転送できること。
*   **F-02-02 瞬間転送:** 距離に関わらず、同一Tick内で搬入から搬出までが完了すること。
*   **F-02-03 Pub/Subモデル:** 送信側(Publisher)と受信側(Subscriber)が互いを知らずとも、チャンネルを介してアイテムが流れること。
*   **F-02-04 フィルタリング:** ホワイトリスト/ブラックリスト、NBT一致、タグ一致、Mod一致による搬入出制限が可能であること。

### F-03: 外部連携機能 (Integration)
*   **F-03-01 標準API:** NeoForgeの `IItemHandler` (Capabilities) を実装し、バニラチェストや他Modのパイプと接続可能であること。
*   **F-03-02 AE2連携:** Applied Energistics 2 の `IMEInventory` として振る舞い、AE2ネットワークから本Modの在庫を読み書きできること。
*   **F-03-03 インベントリキャッシュ:** 頻繁な外部アクセスに対し、Wasmを経由せずにJava側のキャッシュで応答する仕組みを持つこと。

### F-04: 仮想生産機能 (Virtual Production)
*   **F-04-01 Mobシミュレーション:** Mobをスポーンさせず、エネルギー消費のみでLootTableに基づくドロップ品を生成すること。
*   **F-04-02 農業・採掘:** 農作物の成長や鉱石の採掘を、物理ブロックの設置・破壊なしに数値計算のみで行うこと。
*   **F-04-03 コスト自動計算:** 対象のLootTableを解析し、生産にかかるエネルギーコストを自動算出すること。

### F-05: スクリプト・GUI機能 (Scripting & UI)
*   **F-05-01 ユーザー定義ロジック:** JavaScript (Boa) または Rust Script (Rhai) を用いて、アイテムのルーティングや生産制御をプログラム可能であること。
    *   **注:** JavaScript内でのアイテム個数は `BigInt` (任意精度整数) として扱い、`number` 型による精度落ち（2^53超え）を回避すること。
*   **F-05-02 HTML/CSS GUI:** HTML/CSS で記述されたレイアウト定義を解析し、ゲーム内GUIとして描画すること。
*   **F-05-03 外部開発:** VS Code等の外部エディタで編集したスクリプトを、ホットリロードまたはGit連携で反映できること。

---

## 1.3. 非機能要件 (Non-Functional Requirements)

### NF-01: パフォーマンス (Performance)
*   **NF-01-01 低レイテンシ:** アイテム挿入・抽出処理において、Java側のGC発生頻度を「ほぼゼロ」に抑えること（プリミティブ型のみを使用）。
*   **NF-01-02 CPU効率:** 大量輸送時（10,000アイテム/tick）のCPU負荷が、同等のバニラホッパー輸送の 1/100 以下であること。
*   **NF-01-03 キャッシュ効率:** Wasm内データ構造を SoA (Structure of Arrays) とし、CPUキャッシュミス率を最小化すること。

### NF-02: 信頼性・安全性 (Reliability & Safety)
*   **NF-02-01 クラッシュ分離:** Wasmモジュール内でパニック（エラー）が発生しても、Minecraftサーバー全体をクラッシュさせず、当該ネットワークのみを再起動すること。
*   **NF-02-02 データ整合性:** 異常終了時でも、ストレージ内のアイテムデータが消失・破損しないこと（WALまたは二重バッファリング）。
*   **NF-02-03 リソース制限:** ユーザー記述のスクリプトが無限ループしても、命令数制限（Fuel）により強制停止できること。

### NF-03: 互換性 (Compatibility)
*   **NF-03-01 環境:** NeoForge 1.21.1 / Java 21 で動作すること。
*   **NF-03-02 ランタイム:** ユーザーに追加のインストール作業（Rust環境やDLL等）を要求せず、Mod単体で動作すること（Chicory採用）。

### NF-04: 拡張性 (Scalability)
*   **NF-04-01 マルチスレッド:** 複数の独立した輸送ネットワークが存在する場合、Javaの `ExecutorService` を用いて並列処理可能であること。
*   **NF-04-02 モジュール構造:** 「倉庫機能」と「生産機能」が内部的に分離されており、将来的な機能追加や切り離しが容易であること。
*   **NF-04-03 グローバルID共有:** アイテムIDおよびNBT IDのマッピングはワールド全体で共有・統一し、ネットワーク間の統合やアイテム移動時にID変換コストが発生しないようにすること。
