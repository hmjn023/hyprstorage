# 2. Use Case Definition (ユースケース定義書)

## 2.1. アクター定義 (Actors)

| アクター | 説明 |
| :--- | :--- |
| **Player** | ゲーム内でModを利用する一般ユーザー。 |
| **Admin** | サーバー管理者。設定変更やデバッグ権限を持つ。 |
| **External Mod** | パイプや搬出入バスなどを通じて接続してくる他Modのシステム。 |
| **Wasm Runtime** | システム内部でロジックを実行する自律エージェント。 |

---

## 2.2. 主要ユースケース (Primary Use Cases)

### UC-01: ネットワーク構築 (Network Setup)
*   **概要:** プレイヤーが最初の倉庫システムを構築する。
*   **フロー:**
    1.  プレイヤーは `Hyper Core` ブロックを設置する。
    2.  `Hyper Core` は内部で新しいWasmインスタンスを初期化し、ユニークな `NetworkID` を発行する。
    3.  プレイヤーは `Network Connector` (ケーブル) を用いて、チェストや機械を接続する。
    4.  接続された機器は自動的に `NodeID` を割り当てられ、ネットワークの一部となる。

### UC-02: アイテム輸送設定 (Transport Configuration)
*   **概要:** 特定のアイテムをA地点からB地点へ送る設定を行う。
*   **フロー:**
    1.  プレイヤーは「搬入バス (Importer)」の設定画面を開く。
    2.  「チャンネル: 1 (鉱石)」を設定し、フィルタに「鉄の原石」を登録する。
    3.  プレイヤーは「搬出バス (Exporter)」の設定画面を開く。
    4.  「チャンネル: 1」を購読 (Subscribe) する設定を行う。
    5.  システムは搬入された鉄の原石を即座に搬出バスへ転送する。

### UC-03: スクリプトによる制御 (Scripting Control)
*   **概要:** 複雑な条件でアイテムの流れを制御する。
*   **フロー:**
    1.  プレイヤーは `Script Controller` ブロックを設置し、ネットワークに接続する。
    2.  GUIを開き、内蔵エディタまたはVS Codeで以下のロジックを記述する。
        ```js
        // 鉄インゴットが1000個を超えたら、発電機への供給を止める
        if (Network.getItemCount('minecraft:iron_ingot') > 1000) {
            Network.setChannelActive(2, false); // Channel 2: Generator Supply
        }
        ```
    3.  「Save & Run」を押すと、コードがコンパイルされ、Wasm内で毎Tick実行される。

### UC-04: 仮想生産ラインの構築 (Virtual Production)
*   **概要:** ゾンビのドロップ品を自動生産する。
*   **フロー:**
    1.  プレイヤーは `Data Scanner` でゾンビを攻撃し、データを採取する（100%になるまで）。
    2.  採取したデータを `Simulation Chamber` にセットする。
    3.  システムはバックグラウンドでLootTableを解析し、コスト（例: 500 FE/tick）を提示する。
    4.  プレイヤーがエネルギーを供給すると、生産が開始され、結果がネットワークに直接搬入される。

### UC-06: ネットワークの統合 (Network Merging)
*   **概要:** 2つの独立したネットワークをケーブルで接続し、1つの巨大なネットワークにする。
*   **フロー:**
    1.  プレイヤーがネットワークAとネットワークBをケーブルで接続する。
    2.  システムは自動的に「規模が大きい方（または古い方）」を親ネットワークとして選定する。
    3.  子ネットワークの全アイテム在庫とノード設定を親ネットワークにマージする。
    4.  **重要:** アイテムIDはワールド全体で共有されているため、ID変換は不要。単なる加算処理のみで完了する。
    5.  子ネットワークのWasmインスタンスは破棄される。

### UC-E-03: チャンクアンロードへの対応
*   **トリガー:** 接続されているチェストがあるチャンクがアンロードされる。
*   **フロー:**
    1.  Java側で `ChunkUnloadEvent` を検知する。
    2.  当該チャンクに含まれるノードIDをリストアップし、Wasmに `set_node_active(id, false)` を通知する。
    3.  Wasmはルーティング計算からそのノードを除外する（アイテムを送らない）。
    4.  再度チャンクがロードされたら `active=true` に戻す。

### UC-E-01: Wasmクラッシュからの復帰
*   **トリガー:** バグを含むスクリプトや想定外のデータによりWasmがパニックを起こす。
*   **フロー:**
    1.  Java側で例外を検知する。
    2.  プレイヤーに「Network Error: Rebooting...」と通知する。
    3.  直前の正常なデータスナップショットからWasmメモリを復元し、再起動する。
    4.  問題のあったスクリプトは自動的に無効化(Disabled)される。

### UC-E-02: 搬出先満杯 (Overflow)
*   **トリガー:** 搬出先のチェストがいっぱいでアイテムが入らない。
*   **フロー:**
    1.  Wasmは搬出処理を試みるが、失敗としてカウントされる。
    2.  当該ノードを「Smart Sleep」モードに移行させる。
    3.  **起床条件:**
        *   **外部要因:** 隣接ブロックからの更新通知 (`NeighborChanged`)。
        *   **内部要因:** 指数バックオフによる定期チェック (1秒 -> 2秒 -> 最大60秒)。
    4.  GUI上でそのノードが赤色点滅し、詰まっていることをプレイヤーに知らせる。
